01

The answer people will ask for

Say this in 30 seconds

"I use FIFA's public competition endpoints as the official source. A Supabase Cron job calls my Edge Function every minute. That function downloads all 104 World Cup fixtures, validates them, maps FIFA teams to my database teams, and upserts statuses and scores into Postgres. My React app reads the cached results from Supabase every minute, so users do not independently hammer the main FIFA schedule endpoint. For a live match's minute and detailed stats, the browser fetches FIFA's match-specific live endpoints on demand. Once a result is verified, Postgres database triggers score the predictions and update the leaderboard."

Important wording: call it near-live, not millisecond real-time. The central result poll and frontend refresh both run every 60 seconds, so normal score latency is roughly under one to two minutes. Live minute data is also refreshed once per minute.

03

How a score reaches the screen

Supabase Cron fires every minute

* * * * * calls the protected sync-results Edge Function using an API secret stored in Supabase Vault.

The Edge Function calls FIFA

It requests api.fifa.com/api/v3/calendar/matches for competition 17, season 285023, across the complete tournament dates.

It rejects suspicious data

The response must contain exactly 104 fixtures. Unknown FIFA fixture IDs cause failure rather than silently corrupting the database.

It normalizes and maps data

FIFA status numbers become scheduled, live, finished, etc. Team names are normalized and matched through aliases.

It upserts the matches table

Kickoff, teams, score, winner, status, source payload, verification state and sync timestamp are stored in Postgres.

Database triggers score predictions

When a match changes to finished and verified, PostgreSQL calculates every affected user's points and recalculates leaderboard totals.

The browser refreshes Supabase

useWorldCupBackend() reads matches and leaderboard on load and every 60 seconds, then React re-renders the UI.

04

The two FIFA data paths

A. Persistent official result path

FIFA calendar API -> Edge Function -> matches table -> React. This is the authoritative path for scores, match status, winners, prediction locking and scoring.

B. On-demand detail path

Browser -> FIFA live endpoint and FIFA stats endpoint -> match modal. This supplies minute, venue, weather, referee, attendance, events, lineups and statistics when a card is opened.

Database mental model

teams

48 teams, group IDs and aliases used to safely match FIFA naming variations.

matches

All 104 fixtures, official timing, teams, score, winner, status and source metadata.

profiles

Public nickname, avatar seed and favourite team linked to Supabase Auth user ID.

predictions

Group score predictions with points, grade and scoring timestamp.

knockout_predictions

Predicted winner for each knockout match, with 2-point scoring.

leaderboard

Precalculated totals so ranking is fast and does not recompute every page load.

result_sync_runs

Operational log for each FIFA sync: success/failure, received fixtures and timing.

result_corrections

Audit log if FIFA later changes an already verified official result.

Public views

Safe joined data for leaderboard and post-kickoff community predictions.

06

Prediction lifecycle

User signs in

Supabase Auth creates the account. A database trigger automatically creates the matching profile and leaderboard row.

User enters a complete score

The frontend performs an upsert using the unique pair (user_id, match_id), so changing a prediction updates the same row.

Database checks ownership and deadline

RLS ensures the user writes only their own row. A trigger checks server time and rejects writes when status is not scheduled or kickoff has passed.

At kickoff the prediction locks

The client disables editing for UX, but the real security is the Postgres trigger. Changing a laptop clock or calling the REST API cannot bypass it.

After kickoff community picks unlock

RLS permits public reading only after the official kickoff boundary. Before kickoff, users can see their own prediction but not anyone else's.

After verification the grade appears

Community views expose points and result grade only when the match is both finished and verified.

07

Scoring and leaderboard

3Exact group score
1Correct win/draw/loss outcome
2Correct knockout winner
0Wrong prediction

Why scoring is in Postgres

Users cannot award themselves points. Only the database trigger reacts to a verified official match update and writes the grade and points.

How ranking stays quick

private.recalculate_leaderboard() sums group and knockout points only for affected users and updates one leaderboard row per user.

08

How the What If simulation works

Real results replace guesses

buildPredictionScores() uses a user's prediction before kickoff, but uses the official score after the match starts and a real score exists. That preserves the original handwritten-notebook idea.

Tables are calculated in React

calculateTournament() applies points, goal difference, goals scored and mini-table tie logic for all 12 groups and the best eight third-place teams.

Knockout routing is deterministic

buildKnockout() uses official group seeds, FIFA Annex C third-place routing, and winners from previous matches to build the path.

Bad bracket states are prevented

A user cannot pick a team not present in a matchup. Changing an earlier winner clears dependent later-round picks. Clicking the selected winner again removes it.

12

Likely questions and strong answers

Is this scraping Google?

No. The main source is FIFA's own JSON endpoints. The system reads structured API responses, not Google search pages or HTML.

Is it truly real-time?

It is near-live polling. Supabase imports official results every minute and the frontend refreshes every minute. This is simpler and cheaper than maintaining WebSocket infrastructure, and it is appropriate for a small free project.

Why store FIFA data in Supabase instead of calling FIFA from every visitor?

Central caching gives every user the same official state, reduces duplicate provider traffic, enables scoring triggers, keeps an audit trail and avoids making correctness depend on each visitor's browser.

Why are match details fetched directly?

They are optional and opened on demand. Persisting every event, lineup and statistic would add database complexity and writes. The authoritative score still comes through the centralized backend.

What if FIFA is unavailable?

The previous Supabase result remains visible. The sync run is marked failed, the worker retries on the next minute, and match-details UI displays a graceful unavailable state. No fabricated result is inserted.

Can a user change the system clock and submit late?

No. The frontend check is only convenience. The database trigger uses Supabase/Postgres server time and rejects the write independently.

How do laptop and mobile share predictions?

Predictions are stored against the Supabase Auth user UUID, not localStorage or a device ID. Signing into the same account loads the same rows on any device.

How are points protected?

Users submit only predicted values. Database logic resets scoring fields and only the verified official-result trigger calculates points and updates leaderboard totals.

How does community prediction privacy work?

Before kickoff, RLS allows a user to read only their own row. After kickoff, the policy permits locked predictions to be read publicly. Grades stay hidden until official verification.

What is the biggest external risk?

FIFA's public endpoints do not provide this hobby project a contractual uptime guarantee. The design reduces that risk through database caching, strict validation, retries and correction auditing, but it cannot guarantee FIFA will never change an endpoint.

What part is your original product idea?

The product concept is the digital evolution of your 2022 handwritten process: record real results as the tournament progresses, simulate alternate paths, replace guesses with reality after kickoff, and compare predictions with friends. The engineering implements that concept.
